# AGENTS.md — Safe Agent Rules for hermes_compair

This repository contains research, plans, architecture notes, and eventual implementation work for a construction-sector project intelligence system. The intended product is a “project brain” that can ingest project files and communications, extract structured knowledge, build a knowledge graph, maintain timelines and tasks, and propose updates from incoming documentation.

Agents working in this repo must prioritize safety, privacy, auditability, and verifiable outputs over speed.

---

## 1. Core Mission

When working in this repo, help build and document a system that:

- extracts information from construction project files and communications;
- preserves source citations for every extracted fact;
- supports knowledge graph and timeline/task views;
- handles contracts, emails, minutes, drawings, schedules, and notes carefully;
- avoids hallucinated project facts;
- protects confidential project data;
- makes timeline/task updates traceable and reviewable.

---

## 2. Non-Negotiable Safety Rules

### Do Not Invent Project Facts

Never fabricate:

- contract obligations;
- dates or deadlines;
- responsible parties;
- drawing revisions;
- task owners;
- meeting decisions;
- risks;
- project names;
- stakeholder names;
- document contents.

If a fact is not present in provided files or verified tool output, label it as an assumption or ask for source material.

### Citations Are Required for Extracted Facts

Any extracted fact intended for the knowledge graph, timeline, task database, or contract-obligation store must include source evidence:

- file path or source system;
- document title;
- page number, paragraph, row, slide, or email ID when available;
- extraction timestamp;
- confidence level;
- model/tool used if generated by AI.

Do not write extraction logic that stores important facts without evidence fields.

### Human Approval for High-Impact Changes

Do not design or implement silent automatic updates for:

- contractual obligations;
- milestone dates;
- deliverable deadlines;
- payment terms;
- scope changes;
- legal notices;
- liability/penalty clauses;
- tasks assigned to real people from ambiguous language.

High-impact changes must be represented as proposed updates requiring human approval unless the user explicitly changes this requirement.

### Preserve Confidentiality

Construction project data may include confidential contracts, tenders, personal information, pricing, claims, and legal disputes.

Agents must not:

- upload real project files to external services unless explicitly authorized;
- paste large confidential documents into cloud LLMs without approval;
- commit real secrets, credentials, access tokens, cookies, or private project documents;
- weaken access-control assumptions;
- remove privacy/audit safeguards for convenience.

If cloud model use is discussed, document the privacy implications and alternatives.

### Prefer Local-First and Least-Data Model Calls

Default assumption: process documents locally where possible, then pass only necessary snippets to models.

Preferred pattern:

1. Parse files locally.
2. Extract metadata and text locally.
3. Chunk documents with source references.
4. Retrieve only relevant chunks.
5. Pass minimal relevant context to the model.
6. Store model outputs as proposed, cited facts.

Avoid whole-repository or whole-project upload patterns unless explicitly approved.

## 4. File and Data Handling Rules

### Before Processing Files

Agents must identify:

- file type;
- whether the file appears confidential;
- whether it contains personal data;
- whether it is scanned/OCR-heavy;
- whether it is a contract, tender, legal notice, or payment document.

For sensitive file classes, prefer metadata-only handling unless the user authorizes deeper processing.

### When Creating Test Data

Use synthetic examples. Clearly mark them as synthetic.

Good:

- `samples/synthetic_meeting_minutes.md`
- `samples/synthetic_contract_excerpt.md`
- `samples/synthetic_project_schedule.csv`

Bad:

- copied real client minutes;
- real contract clauses;
- real email threads;
- real names or phone numbers.

### When Writing Parsers

Parsers should preserve provenance:

- original file path;
- content hash;
- modification timestamp;
- extraction method;
- page/row/paragraph location where possible.

### When Writing AI Extraction Code

Use schema-constrained outputs. Validate outputs before storing them.

Every extraction schema should support:

- `source_refs`;
- `confidence`;
- `requires_review`;
- `extraction_notes`;
- `raw_evidence_text` or evidence location.

## 5. Model-Use Policy

### Acceptable Uses

Models may be used for:

- document classification;
- structured extraction from approved snippets;
- summarization with citations;
- relationship extraction;
- draft timeline update proposals;
- task/action item proposal;
- risk/issue proposal;
- natural-language Q&A over retrieved chunks.

### Restricted Uses

Models must not be treated as authoritative for:

- legal interpretation;
- final contract obligations;
- final delay/liability conclusions;
- payment decisions;
- safety-critical construction decisions;
- direct timeline/task mutation without review.

### Fine-Tuning Guidance

Do not recommend fine-tuning as the primary memory mechanism for project facts.

Use:

- databases for canonical state;
- vector/RAG indexes for retrieval;
- graph databases for relationships;
- audit logs for provenance.

Fine-tuning may be considered later for extraction style or classification, but not for storing changing project knowledge.

## 6. Development Standards

### Planning First

For non-trivial implementation, create or update a plan in `docs/` before changing code.

Plans should include:

- goal;
- assumptions;
- security/privacy implications;
- data model impact;
- exact files to create/modify;
- tests and verification steps;
- rollback considerations.

### Test-Driven Where Possible

When implementing code:

1. write a failing test or fixture;
2. implement the smallest useful change;
3. run tests;
4. document verification output.

### Keep Prototypes Honest

Do not create fake integrations that look production-ready. If something is a stub, label it clearly.

Use names like:

- `prototype_...`
- `mock_...`
- `synthetic_...`
- `example_...`

### Avoid Over-Engineering

Start with one project, read-only ingestion, citations, and human-approved updates. Do not build multi-tenant enterprise complexity until required.

---

## 7. Git and Commit Hygiene

Before committing:

- run relevant tests or document why they cannot run;
- check that no secrets are staged;
- check that no real confidential documents are staged;
- verify generated files are intentional;
- keep commits focused.

Recommended commit style:

- `docs: add project brain research plan`
- `feat: add document registry schema`
- `test: add synthetic meeting minute extraction case`
- `chore: configure repo metadata`

Never commit:

- `.env` files;
- API keys;
- access tokens;
- raw client data;
- private emails;
- unapproved construction documents.

## 9. Definition of Done for Agent Work

A task is not done until:

- requested files are created or modified;
- outputs are grounded in provided data or clearly marked as assumptions;
- sensitive-data risks are considered;
- relevant tests/checks are run when applicable;
- final response states exactly what changed and where;
- blockers are reported honestly.

---

## 10. Default Assumptions

Unless the user says otherwise:

- treat all real construction project files as confidential;
- do not send full documents to cloud LLMs;
- use local-first parsing;
- require citations for project facts;
- require human approval for timeline/task/contract updates;
- use synthetic data for samples and tests;
- build the MVP for one project before many projects.
