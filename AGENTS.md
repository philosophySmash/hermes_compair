# AGENTS.md for hermes_compair

This repository contains research, plans, architecture notes, schemas, tests, and eventual prototype code for a construction-sector project intelligence system.

The intended product is a project brain that can ingest construction project files and communications, extract structured knowledge, build a knowledge graph, maintain deliverable timelines and tasks, and propose updates when new emails, meeting minutes, notes, contracts, drawings, schedules, and other documentation arrive.

Agents working in this repo must prioritize safety, privacy, auditability, verifiable outputs, and maintainable implementation over speed.

---

## 1. Core Mission

When working in this repo, help build and document a system that:

- extracts information from construction project files and communications;
- preserves source citations for every extracted fact;
- supports knowledge graph and timeline/task views;
- handles contracts, emails, minutes, drawings, schedules, and notes carefully;
- avoids hallucinated project facts;
- protects confidential project data;
- makes timeline, task, and graph updates traceable and reviewable;
- starts with a simple read-only MVP before adding automation that mutates project state.

---

## 2. Operating Mode

Optimize for low token usage while maintaining implementation quality.

Default operating rules:

- Identify the smallest relevant file set before editing.
- Inspect only files directly relevant to the task.
- Prefer small, local changes over broad refactors.
- Make the smallest safe change that satisfies the request.
- Validate with the narrowest relevant check.
- Summarize changed files, validation performed, and remaining risks.
- Preserve existing behavior unless the user explicitly requests a behavior change.
- Choose designs that are easy to maintain and extend when there are multiple equally simple options.
- Do not add complexity for speculative future features.
- If a simple solution is not possible, explain why before using a more complex approach.

Agents should be concise, but never omit safety, privacy, citation, or verification details needed for this repo.

## 3. Non-Negotiable Safety Rules

### Do Not Invent Project Facts

Never fabricate:

- contract obligations;
- dates or deadlines;
- responsible parties;
- drawing revisions;
- task owners;
- meeting decisions;
- risks;
- project names;
- stakeholder names;
- document contents.

If a fact is not present in provided files or verified tool output, label it as an assumption or ask for source material.

### Citations Are Required for Extracted Facts

Any extracted fact intended for the knowledge graph, timeline, task database, contract-obligation store, or project state must include source evidence:

- file path or source system;
- document title when available;
- page number, paragraph, row, slide, drawing reference, or email ID when available;
- extraction timestamp;
- confidence level;
- model or tool used if generated by AI;
- raw evidence location or evidence text where appropriate.

Do not write extraction logic that stores important facts without evidence fields.

### Human Approval for High-Impact Changes

Do not design or implement silent automatic updates for:

- contractual obligations;
- milestone dates;
- deliverable deadlines;
- payment terms;
- scope changes;
- legal notices;
- liability or penalty clauses;
- safety-critical construction decisions;
- tasks assigned to real people from ambiguous language;
- project state changes that affect reporting, responsibility, cost, schedule, or legal exposure.

High-impact changes must be represented as proposed updates requiring human approval unless the user explicitly changes this requirement.

### Preserve Confidentiality

Construction project data may include confidential contracts, tenders, personal information, pricing, claims, legal disputes, designs, schedules, and proprietary drawings.

Agents must not:

- upload real project files to external services unless explicitly authorized;
- paste large confidential documents into cloud LLMs without approval;
- commit real secrets, credentials, access tokens, cookies, or private project documents;
- commit private emails or raw client data;
- weaken access-control assumptions;
- remove privacy, approval, or audit safeguards for convenience.

If cloud model use is discussed, document privacy implications and local or redacted alternatives.

### Prefer Local-First and Least-Data Model Calls

Default assumption: process documents locally where possible, then pass only necessary snippets to models.

Preferred pattern:

1. Parse files locally.
2. Extract metadata and text locally.
3. Chunk documents with source references.
4. Retrieve only relevant chunks.
5. Pass minimal relevant context to the model.
6. Store model outputs as proposed, cited facts.
7. Require review for high-impact facts or low-confidence extractions.

Avoid whole-repository or whole-project upload patterns unless explicitly approved.

### No Em Dash in User-Facing Text

Do not introduce em dash characters, including Unicode U+2014, into user-facing strings, generated reports, UI text, CLI output, or textual displays produced by this application.

Hyphens are fine.

It is acceptable for the system to ingest or display em dashes that already exist in external source documents, but agents must not add new em dashes to generated user-facing text.

## 5. File and Data Handling Rules

### Before Processing Files

Agents must identify:

- file type;
- whether the file appears confidential;
- whether it contains personal data;
- whether it is scanned or OCR-heavy;
- whether it is a contract, tender, legal notice, payment document, schedule, or drawing;
- whether it may be an outdated revision.

For sensitive file classes, prefer metadata-only handling unless the user authorizes deeper processing.

### When Creating Test Data

Use synthetic examples. Clearly mark them as synthetic.

Good examples:

- `samples/synthetic_meeting_minutes.md`
- `samples/synthetic_contract_excerpt.md`
- `samples/synthetic_project_schedule.csv`
- `samples/synthetic_email_thread.eml`

Bad examples:

- copied real client minutes;
- real contract clauses;
- real email threads;
- real names, addresses, phone numbers, or pricing.

### When Writing Parsers

Parsers should preserve provenance:

- original file path;
- source system when available;
- content hash;
- modification timestamp;
- extraction method;
- page, row, paragraph, slide, drawing, or email location where possible;
- detected document revision where possible.

### When Writing AI Extraction Code

Use schema-constrained outputs. Validate outputs before storing them.

Every extraction schema for project facts should support:

- `source_refs`;
- `confidence`;
- `requires_review`;
- `extraction_notes`;
- `raw_evidence_text` or evidence location;
- `extracted_at`;
- `extraction_method`;
- `review_status`.

## 6. Model-Use Policy

### Acceptable Uses

Models may be used for:

- document classification;
- structured extraction from approved snippets;
- summarization with citations;
- relationship extraction;
- draft timeline update proposals;
- task and action item proposals;
- risk and issue proposals;
- natural-language Q&A over retrieved chunks.

### Restricted Uses

Models must not be treated as authoritative for:

- legal interpretation;
- final contract obligations;
- final delay or liability conclusions;
- payment decisions;
- safety-critical construction decisions;
- direct timeline or task mutation without review.

### Fine-Tuning Guidance

Do not recommend fine-tuning as the primary memory mechanism for project facts.

Use:

- databases for canonical state;
- vector or RAG indexes for retrieval;
- graph databases for relationships;
- audit logs for provenance.

Fine-tuning may be considered later for extraction style or classification, but not for storing changing project knowledge.

## 7. Implementation Simplicity

Keep implementation as simple, direct, and robust as possible.

Rules:

- Use existing patterns in the repo before introducing new abstractions.
- Do not add new dependencies, frameworks, background jobs, caches, global state, or configuration unless necessary.
- Avoid cleverness, indirection, premature generalization, and speculative extensibility.
- Make the happy path clear and the failure path explicit.
- Preserve existing behavior unless the prompt asks for a behavior change.
- When there are multiple valid approaches, choose the one with the fewest moving parts.
- If future expansion matters, choose the design that is easiest to extend without making the current implementation more complex.
- If a simple solution is not possible, explain why before implementing a more complex one.

## 8. Development Standards

### Planning First

For non-trivial implementation, create or update a plan in `docs/` before changing code.

Plans should include:

- goal;
- assumptions;
- security and privacy implications;
- data model impact;
- exact files to create or modify;
- tests and verification steps;
- rollback considerations.

If the repo later adds agent prompt templates, document their location and expected format in README.md and this file. Do not assume a top-level `plans/` directory unless it exists and is documented.

### Test-Driven Where Possible

When implementing behavior changes:

1. write or update a failing test or fixture;
2. implement the smallest useful change;
3. run the narrowest relevant validation;
4. document verification output.

### Keep Prototypes Honest

Do not create fake integrations that look production-ready. If something is a stub, label it clearly.

Use names like:

- `prototype_...`
- `mock_...`
- `synthetic_...`
- `example_...`

### Avoid Over-Engineering

Start with one project, read-only ingestion, citations, and human-approved updates. Do not build multi-tenant enterprise complexity until required.

## 9. Testing and Validation

Add or update tests for behavior changes.

Preferred validation flow:

- While iterating, run file-scoped tests or the narrowest relevant check.
- Run full backend, frontend, build, or E2E suites only when requested or before final PR validation.
- If tests cannot be run, state why and describe the risk.
- Do not claim validation passed unless it actually ran and produced passing output.

Future tests should verify that extraction code:

- preserves source references;
- does not store important facts without citations;
- flags high-impact timeline, contract, payment, and safety changes for human review;
- handles duplicate files and email threads safely;
- avoids treating old revisions as current without evidence;
- does not leak unauthorized document chunks into retrieval results.

## 10. Permissions

### Allowed Without Prompting

Agents may do the following without asking first:

- read files and inspect project structure;
- search the codebase only to locate files directly relevant to the requested change;
- edit files directly related to the requested change, only when not on `main`;
- run file-scoped tests and lint checks;
- add or update tests for changed behavior;
- run documented install commands only when dependencies are missing and the task cannot proceed without setup.

### Require Approval First

Agents must ask for approval before:

- editing files while on `main`;
- adding, removing, or upgrading dependencies;
- running full backend, frontend, build, or E2E suites;
- starting long-running services;
- running database migrations against anything other than local or dev;
- deleting files or large code sections;
- changing authentication, authorization, secrets handling, or security-sensitive defaults;
- modifying CI/CD, deployment, infrastructure, or production configuration;
- creating or modifying migrations;
- merging product code;
- changing quality tooling;
- uploading real project documents to external systems;
- using cloud LLMs on real confidential project files.

If the user explicitly asks for one of these actions, treat that as approval for the requested scope only.

## 12. Git and PR Hygiene

Before committing or pushing:

- run relevant tests or document why they cannot run;
- check that no secrets are staged;
- check that no real confidential documents are staged;
- verify generated files are intentional;
- keep diffs small and focused;
- avoid unrelated formatting churn.

Never commit:

- `.env` files;
- API keys;
- access tokens;
- raw client data;
- private emails;
- uploaded test documents with sensitive data;
- local database files;
- unapproved construction documents.

Recommended commit style:

- `docs: add project brain research plan`
- `feat: add document registry schema`
- `test: add synthetic meeting minute extraction case`
- `chore: configure repo metadata`

PR requirements:

- Keep diffs small and focused.
- Include tests for new behavior and regression fixes.
- Update docs or AGENTS.md when commands, structure, setup, or workflows change.
- If a change affects startup flow, environment variables, architecture expectations, or operator workflow, update README.md or relevant architecture docs alongside code.
- Do not include secrets, `.env` contents, sensitive test documents, or local database files.
- Use PR title format: `[area] concise summary`, for example `[api] Add parse job validation`.

Do not merge product code without human approval.
